The Brasa site uses two CSS files that each serve a distinct purpose. Understanding which file controls what makes it straightforward to find and edit any style.

## CSS architecture

Keep structural and reusable styles in `style.css`. Move one-off section styles to `landing.css` to avoid cluttering the core design system.

## Design token system

All visual decisions — colors, typography, border opacities — are declared as CSS custom properties in a single `:root` block at the top of `style.css`.

```css style.css
:root {
  --bg:      #f7f4ef;
  --bg2:     #ede9e2;
  --ink:     #1a1814;
  --ink2:    #4a4640;
  --ink3:    #9a948a;
  --accent:  #c8761a;
  --line:    rgba(26, 24, 20, 0.10);
  --linemed: rgba(26, 24, 20, 0.18);
  --ff-t:    'Cormorant Garamond', Georgia, serif;
  --ff-b:    'DM Sans', sans-serif;
}
```

Every rule in both CSS files consumes these variables. For example, the nav background, card borders, heading typeface, and button color all reference the same token. This means you can retheme the entire site by changing one block.

## Overriding variables for quick theming

To test a theme without editing the main file, add a `<style>` block to `index.html` after the stylesheet links:

```html index.html
<style>
  :root {
    --accent: #2563eb;   /* swap amber for blue */
    --bg:     #ffffff;
    --bg2:    #f1f5f9;
  }
</style>
```

For a permanent theme change, edit the `:root` block in `style.css` directly. See [Colors and fonts](/customization/colors-fonts) for full variable reference.

<Note>
CSS custom properties inherit and cascade like any other property. A variable set on a specific element overrides the `:root` value only for that element and its descendants.
</Note>

## Nav scroll behavior

The navigation bar starts transparent and gains a frosted-glass background once the user scrolls more than 30 px. The scroll listener in `main.js` adds the `.scrolled` class to the `<nav>` element:

```javascript main.js
const nav = document.getElementById('nav');

window.addEventListener('scroll', () => {
  nav.classList.toggle('scrolled', window.scrollY > 30);
}, { passive: true });
```

The CSS in `style.css` controls what the scrolled state looks like:

```css style.css
nav {
  position: fixed;
  top: 0; left: 0; right: 0;
  z-index: 300;
  transition: background .3s, border-color .3s;
  border-bottom: 1px solid transparent;
}

nav.scrolled {
  background: rgba(247, 244, 239, .96);
  backdrop-filter: blur(14px);
  border-color: var(--line);
}
```

To change the scrolled background color, update the `rgba` value in `nav.scrolled` in `style.css`. The value `rgba(247, 244, 239, .96)` is derived from `--bg` (`#f7f4ef`) at 96% opacity.

To adjust the scroll threshold, edit the `> 30` condition in `main.js`:

```javascript main.js
nav.classList.toggle('scrolled', window.scrollY > 30); // change 30 to any pixel value
```

## Reveal animation

Elements with the `.reveal` class start invisible and slide up into view as they enter the viewport. The animation is driven by an `IntersectionObserver` in `main.js` that adds the `.visible` class.

```css style.css
.reveal {
  opacity: 0;
  transform: translateY(16px);
  transition: opacity .7s ease, transform .7s ease;
}

.reveal.visible {
  opacity: 1;
  transform: translateY(0);
}
```

To apply the animation to any new element, add the `reveal` class to it in `index.html`:

```html index.html
<section class="my-section reveal">
  ...
</section>
```

To change the animation duration or easing, edit the `transition` property on `.reveal` in `style.css`.

<Tip>
The IntersectionObserver in `main.js` adds a staggered delay of `i * 60ms` to consecutive elements observed at the same time — so sibling `.reveal` elements automatically cascade without any inline styles needed.
</Tip>

## Responsive considerations

The layout uses CSS Grid throughout. Breakpoints are defined with `@media` queries inside `style.css`. The main responsive patterns are:

<AccordionGroup>
  <Accordion title="Menu layout (sidebar + grid)">
    The `#menu` section uses a two-column grid (sidebar + product grid). On tablet it narrows the sidebar, on mobile it collapses to a single column stack.

    ```css style.css
    #menu {
      display: grid;
      grid-template-columns: 260px 1fr;
    }

    @media (max-width: 1024px) {
      #menu {
        grid-template-columns: 200px 1fr;
      }
    }

    @media (max-width: 768px) {
      #menu {
        grid-template-columns: 1fr;
      }
    }
    ```
  </Accordion>
  <Accordion title="Product card grid">
    The `.menu-grid` inside the main area uses 2 columns by default and collapses to 1 on mobile.

    ```css style.css
    .menu-grid {
      display: grid;
      grid-template-columns: repeat(2, 1fr);
      gap: 3rem 2.5rem;
    }

    @media (max-width: 768px) {
      .menu-grid {
        grid-template-columns: 1fr;
        gap: 2.5rem;
      }
    }
    ```
  </Accordion>
  <Accordion title="About/Nosotros section">
    The `#nosotros` section is a two-column grid (image + content) that stacks on mobile.

    ```css style.css
    #nosotros {
      display: grid;
      grid-template-columns: 1fr 1fr;
    }

    @media (max-width: 768px) {
      #nosotros {
        grid-template-columns: 1fr;
      }
    }
    ```
  </Accordion>
  <Accordion title="Typography scaling">
    Large display headings use `clamp()` to scale fluidly between viewport sizes without hard breakpoints.

    ```css style.css
    .hero-title {
      font-size: clamp(52px, 7.5vw, 108px);
    }

    .sidebar-title {
      font-size: clamp(28px, 3vw, 40px);
    }
    ```
  </Accordion>
</AccordionGroup>

<Warning>
Test on real devices after any grid change. The `object-fit: cover` images in product cards rely on the card's `aspect-ratio: 4/3` — if you change column counts, verify the card images still look correct.
</Warning>
